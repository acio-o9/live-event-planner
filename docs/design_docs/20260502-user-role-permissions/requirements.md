# Requirements: User Role Permissions

**Status**: Draft
**Created**: 2026-05-02
**Developer**: maru

## What（何を作るか）

- ユーザーにグローバルロール（`admin` / `honki_kanrinin` / `user`）を付与する権限システムを実装する
- Prisma スキーマの `User` モデルに `role` カラムを追加し、マイグレーションを行う
- 各 API ルートにロールベースの権限チェックを追加する
- バンドロール（`EventBandMember.role`）との組み合わせで細粒度の操作制御を実現する
- メンバー管理画面で admin / 本気管理人がユーザーロールを変更できるようにする
- `Task.eventBandId`（バンド別タスク）を deprecated にする

## Why（なぜ必要か）

- 現状は認証済みであれば全ユーザーが全操作を行える状態であり、意図しない変更・削除が発生するリスクがある
- ライブイベントの企画・運営は一部の信頼できるメンバー（本気管理人）のみが行うべきであり、権限の明確化が必要

## Requirements（要件）

### Functional Requirements（機能要件）

- [ ] `User` モデルに `role`（`admin` / `honki_kanrinin` / `user`）カラムを追加し、マイグレーションで既存ユーザー全員を `admin` にセットする
- [ ] `lib/permissions.ts` に権限チェック関数を集約し、API ルート全体で再利用できるようにする
- [ ] 各 API ルートにロールチェックを追加する（ライブ作成・編集・削除、バンド管理、マイルストーン・タスク管理など）
- [ ] メンバー管理画面で admin / 本気管理人がユーザーの `role` を変更できる UI および API を実装する
- [ ] `Task.eventBandId` を deprecated とし、API・UI からの新規利用を停止する
- [ ] フロントエンドで権限に応じてボタン・フォームの表示を制御する

### Non-Functional Requirements（非機能要件）

- [ ] 権限チェックは必ずサーバーサイド（API ルート）で行い、フロントエンドの表示制御は補助的な役割とする
- [ ] 権限チェック関数は単体テスト可能な純粋関数として実装する
- [ ] 既存機能への影響を最小化する（破壊的マイグレーションなし）

## Success Criteria（成功基準）

- [ ] ロールに応じた操作制限が全 API ルートで正しく機能する（権限外の操作は 403 を返す）
- [ ] admin / 本気管理人のみがライブイベントを作成・編集・削除できる
- [ ] バンドリーダー・メンバーは自分のバンドの範囲内のみ操作できる
- [ ] 権限チェック関数のユニットテストが通過する
